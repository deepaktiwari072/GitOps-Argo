
# 🚀 Local GitOps Pipeline with Argo CD & NGINX

Welcome to my local GitOps playground! This repository serves as a hands-on guide and documentation for deploying an automated, self-healing **NGINX Web Server** cluster on a MacBook using **Kubernetes (k8s)** and **Argo CD**.

By adopting the **GitOps philosophy**, this project completely eliminates manual infrastructure adjustments (`kubectl apply`). The entire desired state of the application is declared here in Git, and Argo CD continuously works as the "Automator" to synchronize the live cluster status with this repository.

---

## 🛠️ System Architecture



1. **Code Changes:** Kubernetes manifests are updated locally on a MacBook and pushed to this GitHub Repository.
2. **Argo CD Polling:** A single-instance (Non-HA) Argo CD controller running inside the local Kubernetes environment watches this repository.
3. **Automated Reconciliation:** Argo CD automatically detects discrepancies between the repository configurations and the live cluster state, pulling down updates and applying them seamlessly.

---

## 🏗️ Local Environment Setup

This section outlines how the environment was built from the ground up on macOS.

### 1. Local Kubernetes Cluster Activation
Argo CD requires an active Kubernetes API server. This can be launched via Docker Desktop, Minikube, or Kind.
* **Via Docker Desktop:** Navigate to `Settings` ➡️ `Kubernetes` ➡️ Check `Enable Kubernetes` ➡️ `Apply & Restart`.
* **Verification:** Confirm the cluster is accepting connections:
  ```bash
  kubectl cluster-info

  2. Argo CD Core Installation

Install the command-line utility via Homebrew and apply the official upstream lightweight (Non-HA) manifests directly to a dedicated namespace:

# Install the CLI tool
brew install argocd

# Create the dedicated namespace
kubectl create namespace argocd

# Deploy non-HA Argo CD resources
kubectl apply -n argocd --server-side -f [https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml](https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml)


3. Exposing & Accessing the Argo CD Web UI

The server architecture operates internally. Secure connection forwarding bridges it to your Mac's browser:

kubectl port-forward svc/argocd-server -n argocd 8080:443

Access URL: Open https://localhost:8080 (Bypass the self-signed certificate security warnings).

Initial Admin Credentials: The baseline username is admin. Retrieve the autogenerated secret using:
Bash

kubectl -n argocd get secret argocd-initial-admin-secret -o jsonpath="{.data.password}" | base64 -d; echo

4. Terminal Authentication

To issue application controls natively without entering credentials repeatedly, bind the active terminal window session to the cluster:
Bash

argocd login localhost:8080 --insecure

# Initialize repository tracking locally
git init
git add nginx-app.yaml
git commit -m "Add simple nginx app layout"
git branch -M main

# Troubleshooting: Evict old or conflicting remote connections 
git remote remove origin

# Map explicitly to the new GitOps repository using secure SSH strings
git remote add origin git@github.com:deepaktiwari072/GitOps-Argo.git

# Verify targeting routes are clean and secure
git remote -v

# Push branch payload to GitHub tracking systems
git push -u origin main

🤖 Activating the GitOps Automator

Once the manifest code lives safely inside your remote GitHub domain, register the application with Argo CD's polling tracking controllers.

Run the instantiation sequence with Automated Sync policies and Auto-Namespace Creation fully active:
Bash

argocd app create simple-nginx \
  --repo [https://github.com/deepaktiwari072/GitOps-Argo.git](https://github.com/deepaktiwari072/GitOps-Argo.git) \
  --path . \
  --dest-namespace nginx-zone \
  --dest-server [https://kubernetes.default.svc](https://kubernetes.default.svc) \
  --sync-policy automated \
  --sync-option CreateNamespace=true

Key Parameter Highlights:

    --sync-policy automated: Hands scheduling authority over to the engine. Any change on GitHub triggers automatic internal redeployment loops.

    --sync-option CreateNamespace=true: Orders Argo CD to assemble the isolated runtime environment namespace (nginx-zone) if missing.

🔍 Validation, Observability & Troubleshooting Commands

Use these native kubectl tools to check inside the active runtime workspace environments:
Get Live Status Reports

See all instances running under the automated namespace:
Bash

kubectl get pods -n nginx-zone

Comprehensive Component Diagnostics

Review the complete layout of pods, deployments, and services side-by-side:
Bash

kubectl get all -n nginx-zone

Real-time Stream Tracking

Add the watch flag to observe container life-cycles as they transition automatically during a Git push event:
Bash

kubectl get pods -n nginx-zone -w

Checking for Errors (e.g., Image Name Mistakes)

If a typo is accidentally pushed (such as nginx:broken), Kubernetes will flag the status line with ErrImagePull or ImagePullBackOff.
To examine exactly what went wrong under the hood, describe the broken resource:
Bash

kubectl describe pod <INSERT_POD_NAME_HERE> -n nginx-zone

    The GitOps Remediation: Never edit directly via the terminal! Fix the file string inside nginx-app.yaml on your Mac, execute a regular git commit and git push, and watch Argo CD automatically swap out the invalid pods for healthy instances.
